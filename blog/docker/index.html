虚拟化

将一组物理平台，抽象为多组彼此之间互相隔离的计算平台，每一个计算平台都应该拥有冯氏基本部件中的所有设备

CPU的虚拟化

CPU从一开始就是虚拟化的，通过time-sharing技术，每个进程都假设自己拥有完整的CPU 虽然操作方法类似，虚拟机要麻烦很多 首先，每个机器都认为自己拥有内核态和用户态，而内核指令默认可以控制所有的系统调用，也就是说，两台虚拟机之间存在争抢： VMWare是架设在已有的操作系统之上的-Host 每个虚拟机我们称之为Guest Guest是一个进程而已，是一个软件设备，受Host控制 那么，一个Guest上的系统调用，需要：先经过自己虚拟机的内核，再去Host的内核

出于安全考虑，我们需要host的内核运行在环0上，不能让Guest鸠占鹊巢， 可是如果Guest的内核运行在环3上，内核指令没办法使用，因为内核指令都是按照可以直接操控硬件被设计的 所以，每个guest的cpu，实际上是由host的一个进程甚至一个线程模拟出来的cpu 该cpu也有特权指令，只不过是假的 如果真的有特权指令，将会被拦截到host中执行

因此，基于这种方式实现的虚拟化，性能是非常低的 因为特权指令需要两次调度 （BT）另外，在这种情况下，异构的cpu架构是可以被接受的，只不过中间的翻译开销十分惊人 所以，如果Guest和Host的cpu架构相同，那么Guest的用户指令是完全能够直接被运行在hostcpu上的，所以速度相差无几

CPU： 模拟；emulation. — 模拟ring0，1，2，3 虚拟：virtualization —模拟ring0 full virtualization – BT（soft）/HVM（Hardware）

如何提升性能呢？ 硬件虚拟化 增加了一环，环-1 host内核运行在环-1上，特权指令环 Gust内核运行在环0上 HVM …… Hardware VM 环0 - 环-1 的转换由硬件完成，所以提升性能较大

Paravirtualization 半虚拟化 让Guest意识到自己是Guest，自己主动去找Host的内核 Host 改个名字 – VMware Minitor – Hypervisor

Hypervisor 把 CPU 和 Memory 的使用、分配过程虚拟成Hyper call （不包含IO设备，一般而言） 所以guest在使用过程中，如果用到不影响其他guest的不设计内核调用的指令，直接由hypervisor接管 而出现这样指令的时候，被交给hypercall来执行 直接调用、而非翻译

这样的场景中，虚拟机必须知道自己是一个虚拟机 为了实现这种技术，我们必须要对操作系统内核进行修改 半虚拟化是为了提高模拟环0的技术作为补足的

Memory的虚拟化

每一个虚拟机都以为自己能够操作所有的地址

在单机系统中，使用mmu对逻辑地址和物理地址之间进行转换 应用程序malloc时，由内核分配内存给程序使用，而在内核的视角下，因为设计的原因，地址是完整且连续的 在虚拟机的情形下，在内存之上，事实上进行管理的是hypervisor

比如，在虚拟机上一个程序申请使用内存，malloc 虚拟机的内核会为它划分一块地址空间，但是由于这段空间是由hypervisor提供的，本身就是离散的空间， 这就意味着，事实上，当某一进程需要对一段内存进行访问时： 虚拟机内核先去针对该页表mmu寻址，但事实上这个内存并非真的物理内存，所以hypervisor还得把这段被翻译过的地址在翻译一遍，转换到真正的物理地址上去 也就是说，虚拟机上的内存转换，其实是无效的

第一段：应用程序线性地址 – 虚拟机内核地址 ： MMU 第二段：虚拟机内核地址 – host机物理地址 ： 软件模拟shadow_MMU

而且，这样做会带来一个极其严重的后果，TLB很难命中 因为本质是内核完全发生了转变 在单机系统中，无论进程怎么转换，对于同一个物理及来说，逻辑地址和物理地址的转换不会发生冲突 而在虚拟机的情况下，如果第一台虚拟机有如此映射 a – 3 page 当我们换到第二个虚拟机的时候，由于操作系统发生了转变，映射表也会随之变化，这时a — 4 page这种事情几乎必然发生 可是我们会使用tlb来快速寻址，这样的话，我们会得到错误的页面，造成前所未有的影响

为了避免这样的情况发生，每次虚拟机切换就不得不情况tlb，效率会大打折扣

MMU的虚拟化 于是，在硬件上又下了新功夫 ， Intel：EPT， extended page table AMD：NPT， nasted page table TLB虚拟化 tagged TLB，直接标记GVA-HPA

I/O的虚拟化

外存： 硬盘、光盘、U盘

网络设备： 网卡

显示设备： VGA - frame buffer、键盘、鼠标

方式： 模拟：和完全虚拟化没什么太大的区别。- 完全使用软件来模拟真实硬件 半虚拟化 – frontend and backen （通常只能用于硬盘、网卡，不适用于显卡） IO透传： IO-through – 让虚拟机直接使用物理设备（多余的） - hypervisor进行协调（有一块统一的DMA）-IOMMU

鼠标键盘等，使用焦点捕捉，完全软件实现 –临时建立关联方式

从LXC说起

主机级虚拟化：

确实能够实现再一组硬件平台上所谓“跨平台”的互联，不过带给我们的内核开销是巨大的 很多时候，我们创建虚拟机的目的仅仅是为了实现简单的/几个生产应用而已，代价很大 所以需要减少中间层

虚拟的目标是隔离 jail – 保证进程安全的运行 vserver（chroot）

内核通过namespace 对 UTS，Mount，IPC，PID，User，Net对隔离原生支持 加上 chroot ， 才算满足了容器的虚拟 在centos6上不支持 Linux Namespaces

container virtualization 取消主机级模拟的虚拟内核 但是 – 不能提前限制资源

所以，内核必须实现限制资源的功能 Cgroups – control groups 分配资源 划分成组后，对不同的组进行资源分派，组内也可以划分子组进行分配

LXC：LinuX Container 简易整合包 lxc - create， template 完成自动安装发行版以及基本工具的过程

Docker – LXC的增强版 ， 同样，自己并不是容器，只是容器技术的前端而已 通过创建镜像的方式，连接到自己的服务器上，下载镜像，然后启动，极大的简化了虚拟机的使用方式

在一个容器中只运行一个进程，两者之间可以通过容器间通信 以此我可以节省下空间 不过同样会出现问题 ： 调试困难 and 共享内存消失 容器给运维带来了极大的不便，可是反之，给开发人员带来了非常大的便捷 最大的好处在于，环境隔离，部署和分发变得极为便利

另外，通过编排系统，以及外部存储，我可以像是使用进程一般使用docker 所以，我们可以自由的在集群之间进行服务器的调度，因为所有的服务器都可以尽享同样的任务
